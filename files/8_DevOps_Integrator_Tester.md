# 🛡️ ТЗ: DevOps / Integrator / Tester

## Что делаем как команда

Веб-приложение и мобильное приложение **BuildGuard** — система мониторинга нарушений охраны труда на стройке. Загружаешь фото → ИИ находит рабочих без касок и жилетов → результаты попадают в журнал нарушений с дашбордом и отчётами.

Дизайн уже готов (Figma-макеты), фронтенды его реализуют. Бэкенд и ML — отдельные команды.

---

## Что ты делаешь простыми словами

Ты — **связующее звено** проекта. Тебе подчиняется не код, а **процессы**:

1. **Сборка проекта** — чтобы всё запускалось одной командой
2. **Тестирование** — глазами и руками, в роли пользователя
3. **Документация и инструкции** — чтобы любой мог разобраться
4. **Демо-видео и презентация** — главный страховочный план на защиту
5. **Бэкап-задачи** — то, что разработчики не успевают

**Важно:** ты НЕ диктуешь фронтендерам как им работать. Дизайн уже готов, у них чёткое ТЗ. Ты только:
- Тестируешь готовые экраны и **сообщаешь о найденных багах** (через GitHub Issues)
- Предлагаешь улучшения, **которые они могут принять или отклонить**
- Помогаешь, **когда они сами просят**

Твоё рабочее пространство — **корень репозитория, документация, скрипты запуска, тесты, видео**. Ты не лезешь в `frontend-web/src/` или `mobile/screens/` без приглашения.

### Неделя 3: Тестирование API без UI

**Цель:** проверить что сам бэкенд работает корректно — отдельно от фронта.

1. Установи **Bruno** (https://www.usebruno.com — бесплатный аналог Postman) или **Postman**
2. Создай коллекцию запросов — по одному на каждый эндпоинт из `docs/api_contract.md`:
   - `GET /` — health check
   - `POST /detect` — загрузка фото
   - `GET /history` — список

   - `GET /image/{id}` — картинка

3. Сохрани коллекцию в репозиторий: `docs/api_tests.bruno`. Это **полезный артефакт** — другие тоже смогут тестить API.

4. Прогоняй чек-лист API-тестов:
   - [ ] Health check возвращает 200
   - [ ] POST /detect с валидным JPG → возвращает детекции

### Неделя 6: Сборка проекта

**Цель:** сделать так, чтобы любой человек мог запустить проект **за 10 минут**.

**Шаг 1: Финальный README**

Допиши главный `README.md` в корне репо. Должны быть:
- Описание проекта
- Скриншоты или гифки
- Стек технологий
- Архитектурная схема (нарисуй в Excalidraw)
- Инструкция запуска (3 команды максимум)
- Список известных багов и ограничений
- Команда

**Шаг 2: Скрипты автозапуска**

Создай в корне `start.sh` (Mac/Linux):
```bash
#!/bin/bash
echo "Запуск BuildGuard..."

# Бэкенд
cd backend
source venv/bin/activate
uvicorn main:app --host 0.0.0.0 --port 8000 &
BACKEND_PID=$!

# Фронт
cd ../frontend-web
npm run dev &
FRONT_PID=$!

echo "Бэкенд: http://localhost:8000"
echo "Фронт: http://localhost:5173"
echo "Нажмите Ctrl+C для остановки"

trap "kill $BACKEND_PID $FRONT_PID" SIGINT
wait
```

И `start.bat` (Windows):
```batch
@echo off
echo Запуск BuildGuard...
start cmd /k "cd backend && venv\Scripts\activate && uvicorn main:app --port 8000"
start cmd /k "cd frontend-web && npm run dev"
echo Откройте http://localhost:5173
```

**Шаг 3: Зафиксируй версии**

Проверь `requirements.txt` — все версии должны быть с `==1.2.3` (точная версия). Если какая-то библиотека без версии — попроси бэкендера зафиксировать.

То же самое для `package.json` фронта (там это уже делает npm).

**Шаг 4: Тест на чистой машине**

**Самый важный тест.** Возьми ноутбук другого человека (например, PM-а), на котором никто не разрабатывал. Удали ноду и питон если они есть, поставь заново. Открой README и **запусти проект ровно по инструкции**, не отступая ни на шаг.

Если запустилось — ты молодец. Если нет — допиши README, исправь скрипты, проверь снова.

## 🎯 Ключевые понятия

- **Smoke test** — быстрая проверка "вообще запускается ли"
- **Regression test** — проверка, что закрытые баги не вернулись
- **CI/CD** — автоматическая сборка/деплой (для учебного проекта не нужно, но если успеваешь — настрой GitHub Actions с автотестами)
- **Issue** — запись о баге или задаче в трекере
- **Severity** — насколько критичен баг:
  - `critical` — блокирует работу (нельзя запустить, главная функция не работает)
  - `high` — важная функция работает с проблемами
  - `medium` — мелкая функция работает не так
  - `low` — косметика (опечатка, неровный отступ)

## 💡 Почему твоя роль важна

Многие учебные проекты выглядят так:
- Код вроде работает у автора, но никто другой не может запустить
- На защите всё падает, преподаватель не понял что они вообще сделали
- Презентация на коленке за час до защиты

**Ты делаешь так, чтобы у вас этого не было.**

Когда препод спросит "а как это запустить?" — ты показываешь README, по которому всё работает за 10 минут.

Когда сервер падает на защите — ты спокойно включаешь демо-видео, и всё выглядит как продумано.

Когда коллеги говорят "у меня всё работает" — ты единственный, кто **проверил это на чистой машине**.

На защите можешь смело рассказывать о своей роли: интеграционное тестирование, документация, демо. Это серьёзная инженерная работа, и она часто решает 30% впечатления от защиты.
