# 🛡️ ТЗ: QA / DevOps / Designer (универсальный солдат)

## Что делаем как команда

Приложение, которое смотрит на фото со стройки и определяет: есть ли на рабочих каска и жилет. Если кого-то нет — показывает красную рамку и пишет "нарушение".

---

## Что ты делаешь простыми словами

Ты — **последняя линия обороны** проекта. Твоя задача:

1. **Дизайн** — чтобы проект выглядел красиво (не как студенческий)
2. **Тестирование** — найти баги до того, как их найдёт препод
3. **Сборка проекта** — чтобы всё запускалось одной кнопкой по инструкции
4. **Демо-видео и презентация** — главный страховочный план на случай если на защите всё упадёт

Ты **не пишешь код проекта**. Ты делаешь так, чтобы код, который пишут другие, был собран в готовый продукт. Это очень важная роль — без тебя проект работает, но выглядит и презентуется на двойку.

### Неделя 3-5: Тестирование (играешь роль QA)

**Цель:** найти максимум багов до защиты.

Когда фронт и бэк начнут работать вместе — твоё время.

**Шаг 1: Чек-лист тестов**

Создай Google Doc "Тест-кейсы" со списком сценариев:

**Загрузка фото (сайт):**
- [ ] Загрузил обычную JPG-картинку → видны рамки
- [ ] Загрузил PNG → работает
- [ ] Загрузил HEIC (формат iPhone) → работает или внятная ошибка
- [ ] Загрузил GIF → работает или внятная ошибка
- [ ] Загрузил **не картинку** (PDF, doc, txt) → внятная ошибка, сервер не падает
- [ ] Загрузил **огромный файл** (50+ МБ) → ошибка про размер
- [ ] Загрузил картинку **без людей** (пейзаж) → "Объектов не найдено", не падает
- [ ] Загрузил **очень тёмное** фото → есть осмысленный ответ
- [ ] Загрузил **размытое** фото → есть осмысленный ответ
- [ ] Загрузил картинку **с мелкими людьми вдалеке** — что говорит?

**История:**
- [ ] История открывается
- [ ] Записи отсортированы по дате (новые сверху)
- [ ] Превьюшки картинок загружаются
- [ ] Клик по записи → открывается детальная страница
- [ ] На детальной странице видны рамки

**Мобилка:**
- [ ] Камера открывается на iPhone
- [ ] Камера открывается на Android
- [ ] После съёмки фото отправляется на сервер
- [ ] Результат отображается с рамками
- [ ] Работает выбор из галереи
- [ ] Работает на маленьком экране (iPhone SE)
- [ ] Работает на большом (Tablet)

**Сетевые сценарии:**
- [ ] Что если **выключил Wi-Fi** перед отправкой? Появляется ошибка?
- [ ] Что если бэкенд **выключен**? Адекватная ошибка, не вечный спиннер?
- [ ] Что если **очень медленный интернет**? Виден прогресс?

**Шаг 2: Заведение багов**

На каждый найденный баг — тикет в **GitHub Issues**:

```
**Что делал:** загрузил PDF-файл вместо картинки на главной странице

**Что получил:** сервер вернул 500 Internal Server Error, на сайте белый экран

**Что ожидал:** внятное сообщение "Это не картинка, загрузите JPG/PNG"

**Скриншот:** [приложен]

**Где:** Web, Chrome, http://localhost:5173
```

Назначай тикет на ответственного (Frontend/Backend/ML) и ставь label:
- `bug` — что-то сломано
- `ux` — работает, но неудобно
- `polish` — мелочь, можно потом
- `critical` — без этого нельзя сдавать

**Шаг 3: Регресс-тестирование**

Каждые несколько дней:
- Прогоняй чек-лист заново (займёт 30-60 минут)
- Проверяй закрытые баги — точно ли починили
- Если что-то сломалось снова — заводи новый тикет

### Неделя 6: Сборка и инструкции (играешь роль DevOps)

**Главная цель:** сделать так, чтобы любой человек мог запустить проект по инструкции **за 10 минут**.

**Шаг 1: Главный README.md в корне репозитория**

```markdown

Приложение для автоматического обнаружения отсутствия средств индивидуальной 
защиты (касок, жилетов) у рабочих на стройплощадке.

[вставь сюда схему: Frontend ↔ Backend ↔ ML Model]

- **ML**: YOLOv8 (ultralytics)
- **Backend**: Python 3.10+, FastAPI, SQLite
- **Frontend Web**: React + Vite + TypeScript + Tailwind + shadcn/ui

├── backend/          # Сервер FastAPI
├── frontend-web/     # Веб-сайт

## Быстрый старт

### 1. Клонируем репозиторий

```bash
git clone https://github.com/ВАШ_ОРГ/detector-siz.git
cd detector-siz
```

### 2. Запускаем бэкенд

```bash
cd backend
python -m venv venv
source venv/bin/activate  # Windows: venv\Scripts\activate
pip install -r requirements.txt
uvicorn main:app --host 0.0.0.0 --port 8000
```

Сервер запустится на http://localhost:8000  
Документация API: http://localhost:8000/docs

### 3. Запускаем веб-сайт

В новом терминале:

```bash
cd frontend-web
npm install
npm run dev
```

Сайт откроется на http://localhost:5173

### 4. Запускаем мобильное приложение (опционально)

```bash
cd mobile
npm install
npx expo start
```

Скачай **Expo Go** на телефон, отсканируй QR-код.  
⚠️ Телефон и компьютер должны быть в одной Wi-Fi сети.  
⚠️ В файле `mobile/api.ts` укажи IP компьютера (не localhost).

```

**Шаг 2: Зафиксируй версии библиотек**

Без этого через месяц проект перестанет запускаться (зависимости обновятся и сломают код).

В **backend/requirements.txt** должно быть с `==`:
```
fastapi==0.111.0
uvicorn[standard]==0.30.1
python-multipart==0.0.9
ultralytics==8.2.0
pillow==10.3.0
sqlmodel==0.0.18
```

В **frontend-web/package.json** versions уже зафиксированы npm — проверь чтобы не было `^` или `~` перед версиями (или забей, для учебного норм).

**Шаг 3: Скрипты быстрого запуска**

Создай в корне `start.sh` (Mac/Linux):
```bash
#!/bin/bash
# Запускает бэкенд и фронт одновременно

cd backend
source venv/bin/activate
uvicorn main:app --host 0.0.0.0 --port 8000 &

cd ../frontend-web
npm run dev &

wait
```

И `start.bat` (Windows):
```batch
@echo off
start cmd /k "cd backend && venv\Scripts\activate && uvicorn main:app --port 8000"
start cmd /k "cd frontend-web && npm run dev"
```

**Шаг 4: Тест на чистой машине**

Это **самый важный тест**. Возьми ноутбук другого члена команды (на котором проект не разрабатывался). Удали с него всё кроме Node.js и Python. Открой README. Запусти проект **по инструкции, не отступая ни на шаг**.

Если запустилось — ты молодец. Если нет — допиши README, исправь скрипты.

## 🎯 Ключевые понятия

- **QA** (Quality Assurance) — обеспечение качества, тестирование
- **DevOps** — практика, которая объединяет разработку и эксплуатацию (запуск, деплой, мониторинг)
- **Bug** — ошибка в работе программы
- **UX** (User Experience) — то, насколько удобно пользоваться приложением
- **Регресс-тест** — повторная проверка функций, которые раньше работали (вдруг сломалось)
- **Smoke test** — быстрая проверка "вообще запускается ли" без углубления
- **CI/CD** — автоматическая сборка/тестирование/деплой при каждом коммите (для учебного проекта не нужно)

## 💡 Почему твоя роль важна

Многие учебные проекты выглядят так:
- Код вроде работает у автора
- Никто другой не может запустить
- Дизайн ужасный
- На защите всё падает, преподаватель не понял что они вообще сделали

**Ты делаешь так, чтобы у вас этого не было.** Ты — лицо проекта на защите. Когда препод спросит "а как это запустить?" — ты показываешь README, по которому всё работает. Когда сервер падает — ты включаешь видео. Когда коллеги хотят красивый интерфейс — ты даёшь макеты.

На защите можешь смело рассказывать о своей роли: тестировании, документации, демо. Это серьёзная инженерная работа.
